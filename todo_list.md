### 1. 핵심 아키텍처 및 보안 (Core Architecture)
- [ ] **[아키텍처]** 단일 책임 원칙(SRP)에 따라 비대해진 `TradingService`를 `OrderService`(주문), `MarketDataService`(시세/조회), `StreamingService`(웹소켓)로 분리.
- [ ] **[아키텍처]** UI 출력 로직 완전 격리: Service 계층 내부에 존재하는 콘솔 출력(`print`) 로직을 제거하고, View 계층에 위임.
- [ ] **[보안]** 단순 쿠키 기반 인증을 JWT(JSON Web Token) 기반으로 고도화 (세션 만료 및 Secure/HttpOnly 적용).
- [ ] **[BackgroundService]** BackgroundService는 단순 task들의 background 수행만 관리. ForegourndService 만들어서 앞에서 도는 서비스(UseCase)와 분리. Background로 전체종목 data의 종목현재가, ohlcv, bb, ma, rsi 등 변하지 않는 data DB에서 가지고 있도록 수정. 
기존에 있는 service들은 backgroundService(Scheduler), forgroundService(Scheduler)에서 돌수있는 task interface를 가지게 하고, UserAction이 있으면 backgroundService는 suspend/resume 할 수 있도록 수정. 
두가지 스케줄러에 모두 등록되는 Service 도 있을수 있음.
(e.g) foreground: 현재가 조회, 계자잔고 조회, 매수/매도(최우선 우선순위), 랭킹, 시가총액, 모의투자 기록, 전략 스케줄러(전략 실행/정지), 프로그램매매 구독 등 User Action에 의한 동작
backgournd: 수행중인 전략 scheduler(전략에서 수행하는 매수/매도는 forground의 매수매도와 동일한 최우선순위) 랭킹정보 udpate, 전체 종목 정보 update, oneil_service의 poolA Update 등 장마감 이후 한번씩 고정된 data를 db에 올리는 작업 등.

- [ ] **[전략]** 전략의 성능 개선 필요.
  🚀 1. scan() 루프의 순차적 API 호출 병렬화 (가장 높은 성능 향상)
    [현재 문제점]
    scan() 메서드의 for code, item in watchlist.items(): 루프는 종목들을 하나씩 **순차적으로 대기(await)**하며 처리합니다.
    만약 워치리스트에 60개의 종목이 있고, 1종목의 현재가를 조회하는 데 0.1초가 걸린다면 전체 스캔에만 최소 6초가 걸립니다. 종목이 늘어날수록 봇의 반응 속도가 선형적으로 느려집니다.

    [개선 방안: Chunk 기반 asyncio.gather]
    API 호출을 묶어서 동시에 처리(Concurrency)하되, 증권사 API의 초당 호출 제한(TPS, 보통 한국투자증권은 20 TPS)을 넘지 않도록 Chunk(예: 10개씩) 단위로 쪼개서 병렬 처리해야 합니다.
  🧠 2. check_exits() 내 OHLCV 데이터 중복 호출 제거
    [현재 문제점]
    check_exits에서 매도를 검사할 때, 시간 손절(_check_time_stop)과 추세 이탈(_check_trend_break) 메서드가 각각 독립적으로 존재합니다.
    만약 어떤 종목이 시간 손절에 걸리지 않아 다음 로직인 추세 이탈 검사로 넘어가게 되면, 동일한 종목의 일봉(OHLCV) 데이터를 API로 두 번 연속 조회하게 됩니다. 이는 심각한 API 낭비입니다.

    [개선 방안: OHLCV 데이터를 한 번만 조회하여 파라미터로 전달]

  💾 3. 잦은 동기(Synchronous) File I/O의 일괄 처리(Batching)
    [현재 문제점]
    check_exits 루프 안에서 보유 종목의 현재가가 고점을 돌파(current > state.peak_price)할 때마다 즉시 self._save_state()를 호출합니다.
    _save_state는 파이썬의 동기 함수(json.dump)를 사용해 디스크에 파일을 씁니다. 만약 상승장에 5개 종목이 동시에 고점을 돌파하면, 루프 안에서 비동기 흐름을 막고 디스크 쓰기를 5번 연속으로 수행하여 시스템에 랙(Lag)을 유발합니다.

    [개선 방안: Dirty Flag 패턴을 이용한 일괄 저장]
    루프 내에서는 메모리(self._position_state)만 업데이트해 두고, 루프가 완전히 종료된 후 변경 사항이 있었을 때만 딱 1번 디스크에 저장하도록 변경합니다.
